= No-Nonsense NNTPS — Show Me The Receipts
:toc:
:icons: font

This document backs up the README claims with code evidence and honest architectural tradeoffs.

== README Claim 1: Mandatory NNTPS (TLS-Only, No Plaintext Fallback)

[quote, README]
____
The client speaks NNTPS only. No insecure NNTP fallback. No "allow plaintext just this once" checkbox. No pretending transport security is optional.
____

=== How It Works

The security model is enforced at the connection layer (`lib/no_nonsense_nntps/connection.ex`):

1. **Port Hardcoding**: Default port is 563 (NNTPS). Port 119 (plaintext NNTP) is rejected at the module interface level.
2. **TLS Mandate**: Connection module explicitly requires `tls: true` in the OTP socket options. There is no `tls: false` codepath.
3. **Certificate Validation**: TLS options include `:verify => :verify_peer` with default CA bundle. No `:verify => :verify_none` override available to users.
4. **Error Transparency**: If TLS handshake fails (e.g., self-signed cert, expired cert, wrong hostname), the error propagates as a hard `:tls_failed` exception rather than falling back to plaintext.

**Code Evidence:**
- Connection setup at `lib/no_nonsense_nntps/connection.ex` lines 15-35: Elixir `:ssl.connect/3` call with `verify: :verify_peer`
- Port validation at `lib/no_nonsense_nntps/server.ex` lines 8-12: port must be 563; 119 raises `:invalid_port`
- Test: `test/no_nonsense_nntps/connection_test.exs` demonstrates that a mock plaintext connection immediately fails

=== Why This Design

NNTP has aged poorly for security. Usenet news servers are frequently accessed by botnets, spam harvesters, and credential stealers. Mandatory TLS:
- Prevents credential interception on coffee-shop WiFi
- Blocks passive traffic analysis of which groups a user reads
- Enforces proper certificate validation (harder for MITM attackers)

Historical NNTP clients (tin, nn, slrn) have no TLS support or optional TLS, creating a false sense of security for users who think they are encrypted.

=== Honest Caveat: Certificate Pinning Not Implemented

The connection validates against the OS certificate store only. It does **not** pin certificates or public keys. This means:

1. A compromised CA can issue a fake certificate for a news server (rare but possible)
2. If your sysadmin installs a corporate proxy certificate in your trust store, NNTPS traffic is visible to the proxy
3. The client cannot distinguish a legitimate certificate update from a MITM

**Mitigation**: For high-security deployments, users should disable the system CA and manually supply a pinned cert via environment variable (future feature). For typical use, system validation is sufficient.

== README Claim 2: Thread-First UI, Not Feed-First

[quote, README]
____
Discussion is treated as a tree, not a stream. Thread structure, reply relationships, subject drift, and long-form readability are core product concerns rather than incidental implementation details.
____

=== How It Works

The data model (`lib/no_nonsense_nntps/thread.ex`) centers on parent-child relationships:

1. **Thread Root**: Each article has an optional `in_reply_to` header (RFC 5536). Articles with `in_reply_to == nil` are thread roots.
2. **Tree Assembly**: On group open, the client fetches headers and builds a tree structure via `NoNonsenseNntps.ThreadBuilder.assemble/1`:
   - For each article, extract `references` header (space-separated chain of message IDs)
   - Recursively build parent-child links
   - Orphaned articles (missing ancestor) become micro-roots
3. **UI Rendering**: ReScript frontend (`ui/components/ThreadView.res`) renders a collapsible tree:
   - Root articles are top-level cards
   - Replies indent beneath parents
   - Subject lines show threading context ("Re: Original Subject [2 replies]")
   - Sorting options: "by date (thread root first)" or "by depth (longest threads)"
4. **Offline Cache**: Thread structure is serialized to SQLite (`lib/no_nonsense_nntps/cache.ex`) so offline navigation works

**Code Evidence:**
- Thread model: `lib/no_nonsense_nntps/thread.ex` (struct with `id`, `in_reply_to`, `references`, `subject`)
- Tree builder: `lib/no_nonsense_nntps/thread_builder.ex` — implements `assemble/1` (recursive, handles orphans)
- UI component: `ui/components/ThreadView.res` — TEA-style rendering with collapsible tree nodes
- Cache schema: `lib/no_nonsense_nntps/cache.ex` — SQLite table `articles` with JSON tree structure in `thread_ancestors` column

=== Why This Design

Feed-based readers (Slurp, Newsblur) present articles chronologically, losing threading context. You see "Re: Subject" but click to find the original post. This breaks long-form discussion. Thread-first design:
- Surfaces discussion relationships immediately
- Lets readers understand context without hunting
- Enables "skip entire thread" workflows for readers (ignore marketing, read technical discussions)
- Makes subject drift visible (long threads with multiple topics)

=== Honest Caveat: Threading Fails Under Subject Drift & Spam

The thread assembly algorithm has a critical flaw: **it only uses the `references` header**. If a subject line changes mid-thread, or if spam emails are posted with fake references headers, threading breaks:

1. **Subject Drift**: A thread starts as "Q: How to install Erlang?" then evolves into "Re: Performance tuning tips". The same thread, but a reader scanning subjects might miss context.
2. **Fake References**: A spammer posts "Re: Original Article" but fabricates the `references` header. The fake post appears to be part of the legitimate thread.
3. **Missing Headers**: Old articles lack proper `references` headers. They become orphaned threads or miscategorized.

**Current Mitigation**: The UI shows "(orphaned)" labels for articles missing parent ancestors. A future version could use Bayesian filtering on subject lines + dates to repair obvious misthreads, but this is beyond MVP scope.

|===
| Layer | Technology | Reason

| **Protocol & State** | Elixir (BEAM) | GenStateMachine for explicit connection lifecycle; supervision tree for fault tolerance
| **Connection Management** | OTP `:ssl` module | Native Erlang TLS, battle-tested in telecom
| **Storage (Offline Cache)** | SQLite + Ecto | Simple, file-based, no server dependency; Ecto schemas for migration tracking
| **Frontend UI** | ReScript + TEA | Type-safe, compiles to JS; TEA pattern matches Elixir GenServer semantics
| **HTTP Bridge** | Bandit | Pure-Elixir HTTP/1.1 server; avoids Node.js dependency
| **Formal Verification (Future)** | Idris2 (ABI) | For proven message parsing and thread assembly correctness

== How To Verify Claims

=== Test TLS Enforcement

1. Attempt connection to plaintext NNTP server (port 119):
   ```bash
   iex -S mix
   > NoNonsenseNntps.Connection.connect("nntp.example.com", 119)
   ** (RuntimeError) plaintext NNTP (port 119) not allowed — NNTPS required
   ```

2. Attempt connection with self-signed cert (should fail unless pinned):
   ```bash
   > NoNonsenseNntps.Connection.connect("self-signed.local", 563)
   ** (:tls_failed) certificate validation failed — check server cert
   ```

3. Successful NNTPS connection (with valid cert):
   ```bash
   > NoNonsenseNntps.Connection.connect("news.example.com", 563)
   {:ok, %NoNonsenseNntps.Connection{...}}
   ```

=== Test Thread Assembly

1. Run test suite:
   ```bash
   mix test test/no_nonsense_nntps/thread_builder_test.exs
   ```

2. Observe orphan handling:
   - Articles with missing parents are marked `(orphaned)`
   - Tree still renders correctly with multiple micro-roots

3. Load real newsgroup (if you have a test NNTPS server):
   ```bash
   iex -S mix
   > NoNonsenseNntps.ClientManager.connect("news.example.com", 563)
   > articles = NoNonsenseNntps.ClientManager.fetch_group("comp.lang.rust")
   > tree = NoNonsenseNntps.ThreadBuilder.assemble(articles)
   > Enum.map(tree, & &1.id)  # Top-level thread IDs
   ```
