= No-Nonsense NNTPS
:toc:
:toclevels: 3
:sectnums:

A secure, modern NNTP reader for serious text discussion, mailing-list archives, and classic Netnews infrastructure.

No-Nonsense NNTPS is a local-first client for reading newsgroups and NNTP-accessible archives over encrypted transport. It is designed for people who still value threaded discussion, durable archives, offline reading, and protocols that do one thing well.

The goal is not to reinvent Netnews.
The goal is to make it usable, safe, and pleasant again.

== Why This Exists

NNTP still matters in a handful of places:

* traditional Usenet and private news servers
* mirrored mailing-list archives
* long-lived technical discussion systems
* text-first communities that value threading over feeds

Most existing clients are either old, awkward, or built around assumptions that no longer feel acceptable.

No-Nonsense NNTPS is an attempt to provide:

* secure-by-default transport
* a modern reader experience
* strong standards compatibility
* good offline behaviour
* clean interoperability with existing NNTP/NNTPS systems

== Design Principles

=== Secure by Default

This client supports encrypted NNTP transport only.

Plaintext NNTP is intentionally out of scope.
If a server cannot speak secure NNTP, this client will not connect to it.

=== Standards First

No-Nonsense NNTPS aims to be a serious client for existing NNTP infrastructure, not a replacement protocol and not a partial toy implementation.

Compatibility, predictable behaviour, and clear failure modes matter more than novelty theatre.

=== Local First

Messages, threading metadata, and article state should remain useful even when the network is unavailable.

Offline reading is a core feature, not an afterthought.

=== Text Over Noise

This project is built for people who want to read and write messages, follow threads, score content, and manage archives without algorithmic feeds, attention traps, or unnecessary chrome.

== Core Features

Planned and in-progress capabilities include:

* NNTPS-only connections
* standards-compliant NNTP command support
* thread-aware article browsing
* local caching for offline reading
* subscription management for multiple servers
* article scoring, filtering, and killfile-style controls
* fast search over cached articles and headers
* export and archival workflows
* clean handling of modern text formats
* support for archive-oriented NNTP use cases where practical

The emphasis is on reliability and legibility rather than social features, recommendation engines, or protocol churn.

== Non-Goals

This project does not aim to be:

* a social media platform
* a web forum clone
* an all-purpose mail client
* an insecure legacy compatibility layer
* a speculative replacement for NNTP itself

It is a reader.
It should do that job well.

== Architecture

The current implementation direction is:

* *Elixir* for orchestration, connection lifecycle, and state management
* *ReScript* for the client UI
* *TEA-style architecture* for predictable frontend state transitions
* optional integration with the wider hyperpolymath tooling ecosystem where it genuinely improves the software

The architecture is intended to keep protocol logic, storage, UI state, and platform integration clearly separated.

== Project Scope

No-Nonsense NNTPS is focused on the client side:

* connecting to existing NNTPS servers
* synchronising headers and articles
* maintaining local state
* presenting readable, navigable discussion threads

Server implementation is out of scope for this repository.

== FFI and ABI Policy

If this project requires foreign-function boundaries, they should be small, explicit, and justified.

Preferred structure:

....
project/
├── src/
├── ui/
├── native/
├── bindings/
└── docs/
....

Where applicable:

* ABI definitions should be versioned and documented
* generated bindings should not become the public architecture
* native code should exist to support the client, not dominate it
* proof-heavy or formally verified components should be used where they materially reduce risk

The baseline rule is simple:
do not introduce a foreign-function boundary unless it earns its keep.

== User Experience Goals

The ideal user experience is:

* quick to connect
* easy to subscribe and browse
* pleasant for large thread trees
* robust on unreliable networks
* useful when offline
* respectful of keyboard-heavy workflows
* readable for long-form technical discussion

This project should feel more like a serious reader and less like a browser tab pretending to be one.

== Development Status

This project is in early design and implementation.

Current priorities:

. establish a clean protocol core
. define local storage and sync behaviour
. build a usable thread and article model
. provide a minimal working UI
. validate against real NNTPS servers and archive-style workloads

== Roadmap

=== Phase 1: Minimal Viable Reader

* connect to an NNTPS server
* authenticate where required
* list groups
* fetch headers
* fetch and display articles
* persist local state

=== Phase 2: Usable Daily Driver

* subscriptions
* offline cache
* thread navigation
* scoring and filters
* search
* export/import of local state

=== Phase 3: Power-User Features

* richer archive workflows
* advanced filtering
* keyboard-first navigation polish
* better multi-server support
* optional integrations with other hyperpolymath components

== Build Philosophy

This project prefers:

* small understandable components
* explicit state machines
* reproducible builds
* conservative dependencies
* clean failure handling
* documentation that explains trade-offs rather than hiding them

== Summary

No-Nonsense NNTPS is a secure, local-first NNTP reader for people who still want durable threaded discussion without feed logic, insecure defaults, or nostalgia-driven rough edges.

It is not trying to resurrect the whole old world.

It is trying to build one good tool for the part that still matters.
