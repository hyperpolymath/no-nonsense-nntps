= No-Nonsense NNTPS
:toc:
:toclevels: 3
:sectnums:

A secure, local-first, thread-first NNTPS reader for people who still want durable discussion without the noise, clutter, and unsafe defaults of aging clients.

[NOTE]
====
No-Nonsense NNTPS is intentionally *NNTPS-only*.

Plaintext NNTP is not supported.
There is no insecure fallback mode.
====

== Overview

No-Nonsense NNTPS is a modern reader for Network News Transfer Protocol over TLS.

It is built for people who still value structured conversation, predictable tooling, local control, and software that does one job well. The aim is not to reinvent Usenet as a social platform, nor to build a protocol museum piece. The aim is to deliver a serious, comprehensible client with sane security defaults, excellent thread handling, durable local state, and a reading experience that respects long-form discussion.

This project is developed within the broader hyperpolymath ecosystem, but it is first and foremost a usable NNTPS reader.

== Why This Exists

NNTP is old, but the underlying use cases are still real:

* long-lived threaded discussion
* archive-oriented reading
* text-first technical communities
* private or public news servers
* users who want control over local state instead of surrendering everything to a browser tab or feed algorithm

Many existing clients are either:

* capable but visibly dated, awkward, or insecure by modern expectations
* modern-looking but shallow, incomplete, or overly web-shaped

No-Nonsense NNTPS exists to offer another path:

* *secure by default*
* *thread-first rather than feed-first*
* *local-first rather than session-fragile*
* *pleasant for heavy readers*
* *boring in the best possible way*

== Design Principles

=== Security First

The client speaks NNTPS only.

No insecure NNTP fallback.
No “allow plaintext just this once” checkbox.
No pretending transport security is optional.

=== Threads Matter

Discussion is treated as a tree, not a stream.

Thread structure, reply relationships, subject drift, and long-form readability are core product concerns rather than incidental implementation details.

=== Local First

Subscriptions, read state, filters, caches, and preferences should remain understandable and useful even when the network is unavailable or unreliable.

The client should not invent its own reality behind the user’s back.

=== Text First

This project is optimized for reading and writing discussion.

Rich content may be handled where practical, but the software is not built around endless embeds, ornamental chrome, or attention-harvesting interface tricks.

=== Modern Engineering Without Performative Complexity

The project uses contemporary tools where they materially improve correctness, maintainability, or safety.

Technology choices should justify themselves in the product, not merely in architecture diagrams.

=== Interoperate, Do Not Cosplay Revolution

This is a client for real NNTPS infrastructure.

Compatibility, predictable behaviour, and clean failure modes matter more than novelty theatre.

== Core Features

The intended feature set includes:

* NNTPS account configuration
* secure server connection management
* newsgroup discovery and subscription management
* thread tree browsing
* article reading and composition
* reply and follow-up workflows
* durable read/unread tracking
* offline-friendly local caching
* search and filtering
* scoring and killfile-style rules
* attachment and content-handling policies
* practical import/export of user configuration and state where appropriate
* support for archive-oriented workflows where feasible

The emphasis is reliability, legibility, and user trust rather than social features, recommendation engines, or protocol churn.

== Non-Goals

This project is *not* trying to be:

* a social media client
* a web forum clone with NNTP paint on top
* a kitchen-sink showcase for every ecosystem standard at once
* a protocol replacement
* a nostalgia toy that excuses poor ergonomics
* an insecure legacy compatibility layer

It is a reader.
It should do that job well.

== Security Model

No-Nonsense NNTPS takes a narrow and explicit security stance.

=== Transport

* NNTPS only
* TLS is mandatory
* insecure NNTP is rejected by design

=== Credentials

Credential handling should be conservative and boring.

The client should avoid vague trust boundaries, leaky abstractions, and “just put it in a config file” shortcuts. Storage, access, and transmission paths should be explicit.

=== Content Handling

Articles, headers, and attachments must be treated as untrusted input.

Rendering and preview behaviour should aim to reduce surprises, avoid unsafe execution paths, and make policy visible to the user.

=== Operational Intent

The client should make the safe thing easier than the unsafe thing.

== User Experience Goals

A good NNTPS reader should feel:

* calm rather than noisy
* structured rather than feed-driven
* trustworthy when dealing with state
* efficient for keyboard-heavy or high-volume use
* readable over long sessions
* useful offline
* predictable under reconnect and resync

This project cares about ergonomics for serious readers, not just protocol compliance.

== Project Scope

No-Nonsense NNTPS focuses on the client side:

* connecting to existing NNTPS servers
* authenticating where required
* synchronising headers and articles
* maintaining local caches and user state
* presenting readable, navigable threads
* supporting practical reply and posting workflows

Server implementation is out of scope for this repository.

== Architecture

The current implementation direction is:

* *Elixir* for orchestration, protocol sessions, storage coordination, and stateful connection management
* *GenStateMachine* for explicit connection lifecycle handling
* *ReScript* for a type-safe client UI
* *TEA-style state management* for predictable frontend behaviour
* *cadre-tea-router* for route handling in web-based interfaces where applicable

The architecture is intended to keep protocol logic, local state, rendering, and platform integration clearly separated.

The goal is a system that remains understandable under reconnects, sync edge cases, and long-lived sessions.

== Status

This project is in active design and early implementation.

Current priorities are:

. establish a clean NNTPS connection model
. define article, header, and thread data structures
. implement durable local state
. build a minimal thread-first UI
. validate behaviour against real-world servers and realistic reading workflows
